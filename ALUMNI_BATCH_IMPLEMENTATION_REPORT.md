# CampusLink – Alumni Batch Management (Phase 4E) Implementation Report

This report outlines the implementation details for the Alumni Batch Management module of CampusLink, connecting the School Admin Dashboard, the Alumni Joining Flow, and the Public School Profile page.

---

## 1. Database Architecture & Schema

We have added four new tables and extended the `profiles` table to manage alumni batches, invitations, memberships, and requests.

### Table Definitions

1. **`public.alumni_batches`**: Organises alumni by graduating/passing year.
   - Unique Index: `(school_id, passing_year, COALESCE(department, ''), COALESCE(program, ''))` ensures no duplicate batches.
2. **`public.alumni_invites`**: Secure invitation links generated by the admin for batches.
   - Code validation includes status check ('active' / 'disabled') and expiration check.
3. **`public.alumni_members`**: Association table linking users to graduation batches.
   - Constraint: `unique_batch_member` prevents a user from joining the same batch twice.
4. **`public.alumni_requests`**: Joining requests from alumni pending admin review.
   - Statuses: `pending`, `approved`, `rejected`, `info_requested`.

### RLS (Row Level Security) Policies
- Batches, invites, members, and requests are fully protected.
- Only verified School Administrators (checked via school owner id or `school_members` admin/owner roles) and Super Administrators can write/modify data.
- Public read access is granted on `alumni_invites` to resolve invite codes without authentication during join flows.
- Authenticated users can view batches, members, and submit requests for themselves.

### Triggers & Automation
- **`tr_on_alumni_request_status_change`**: Fires when an admin updates request status to `'approved'`. It automatically:
  1. Updates the applicant's profile `user_type` to `'alumni'`, links `school_id`, and assigns `passing_year` and `department`.
  2. Inserts or updates the applicant's record in `school_members` with `role = 'alumni'`.
  3. Inserts a record in `alumni_members` linking them to the target batch.
  4. Increments the `uses_count` on the `alumni_invites` link.

## 2. School Admin Dashboard Integrations

The **Alumni Management** console is nested inside the School Admin Dashboard navigation sidebar under **Community -> Alumni**.

### Sections
1. **Overview & Statistics**:
   - Total Alumni count.
   - Total Batches count.
   - Pending Alumni Requests count (and nav/tab badges).
   - Most Active Batch (calculated by member count).
   - Newest Batch (most recently created).
2. **Graduation Batches**:
   - Lists all school batches with their passing year, program, department, member count, invite status, and created date.
   - Actions: Create Batch, Edit Batch, View Batch Members (including member search), Copy Invite Link, Generate QR Code, Deactivate Link, Delete Batch.
3. **Invite Links**:
   - Details status, uses count, and expiration of each batch code.
   - Actions: Copy URL, Generate QR Code, Deactivate/Activate, Regenerate Invite Link (invalidates old code immediately).
4. **Pending Requests**:
   - Lists requests with fields like name, username, email, and graduation details.
   - Actions: Approve, Reject, Request Info (prompts admin for message and transitions request to `info_requested`).

## 3. Public School Profile (Community Tab)

Instead of a single unstructured list, alumni are organized chronologically by batch on the Public School Profile page.

### Features
1. **Graduation Batches Grid**:
   - Lists graduating batches (e.g. `Batch of 2026 — 48 Members →`).
   - Cards are responsive, interactive, and styled according to CampusLink's premium aesthetics.
2. **Batch Detail Sub-page**:
   - Displays a banner image (using the batch cover image or a default dark linear gradient).
   - Shows statistics: Total Members, Class Representatives, Top Contributors, and Recently Joined (members added in last 30 days).
   - Includes a **Member Search bar** to filter members dynamically by Name, Username, or Department.
   - Interactive member grid linking directly to individual profiles.
   - Clean `← Back to Community` navigation.

## 4. Onboarding Joining Flow

The joining flow is implemented at `join-alumni.html` in a self-contained onboarding card using CampusLink's signature dark background with neon glowing ambient orbs.

### User Journey
1. **Link Verification**: Resolves the invite parameters (`?code=CODE` or `?school=ID&batch=YEAR`). Shows a spinner during resolution. If invalid, displays an error screen.
2. **Auth Gateway**: If not logged in, displays a tab toggle for **Create Account** and **Sign In**.
   - Creating an account requires Name, Email, Username, Password (auto-filling role = Alumni, school, passing year).
3. **Email Verification Alert**: If email confirmation is enabled in Supabase, prompts the user to check their inbox.
4. **Join Request Confirmation**: Once logged in, shows the user details and a "Submit Request" button.
5. **Success State**: Submits request to the database and informs the user their application is pending admin approval.
